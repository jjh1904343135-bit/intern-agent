# Agent / RAG Eval Report

This report is the interview-facing reliability checklist for Qingcheng AI. It is intentionally small and strict: the goal is not high case volume, but proving that Agent behavior, RAG grounding, job recommendation evidence, interview follow-up policy, and resume scoring rubric do not silently regress.

## Dataset

| Dataset | Cases | Purpose |
| --- | ---: | --- |
| `backend/tests/agent_eval/golden_cases/chat_cases.jsonl` | 3 | AI assistant intent, tool choice, and tool argument checks |
| `backend/tests/agent_eval/golden_cases/rag_cases.jsonl` | 1 | Knowledge RAG trigger and grounded context injection |
| `backend/tests/agent_eval/golden_cases/job_match_cases.jsonl` | 1 | Job recommendation explanation and evidence contract |
| `backend/tests/agent_eval/golden_cases/interview_cases.jsonl` | 1 | Interview question plan and follow-up strategy |
| `backend/tests/agent_eval/golden_cases/resume_score_cases.jsonl` | 1 | Resume scoring rubric contract |

Total golden cases: 7

## Metrics

The report is generated by:

```powershell
docker compose run --rm api sh -c "alembic upgrade head && pytest tests/agent_eval -q"
docker compose exec api python -m evals.agent.run_agent_eval
docker compose exec api python -m evals.rag.eval_knowledge_rag --generate-answers
```

The generated report must include these metrics:

| Metric | Meaning | Target |
| --- | --- | ---: |
| Intent Accuracy | Whether the Agent classified the user goal correctly | >= 0.80 |
| Tool Call Accuracy | Whether required tools were selected | >= 0.80 |
| Argument Accuracy | Whether extracted tool arguments match the case | >= 0.80 |
| Job Match Explanation Rate | Whether recommendation output includes evidence fields | >= 0.80 |
| Resume Rubric Compliance | Whether resume score follows `resume_score_v1` | >= 0.80 |
| Interview Follow-up Hit Rate | Whether follow-up policy matches answer quality | >= 0.80 |
| RAG Grounding Rate | Whether RAG context is used and marked as reference material | >= 0.80 |
| RAG Recall@3 / Recall@5 | Whether expected chunks are retrieved in top-k | tracked in `backend/evals/rag/rag_eval_report.md` |
| RAG MRR | Whether correct chunks rank early | tracked in `backend/evals/rag/rag_eval_report.md` |
| Grounded Answer Rate | Whether generated answers are supported by retrieved context | tracked in `backend/evals/rag/rag_eval_report.md` |

## Failure Review Format

Every failed case should be recorded as:

```text
case_id:
  expected:
  actual:
  suspected_layer: intent | tool | argument | rag | job_match | interview | resume_score | prompt_contract
  fix_plan:
```

This keeps the project defensible in an internship interview: we can explain not just "the Agent works", but which behavior is tested, where it failed, and how we narrowed the fix.
