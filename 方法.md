记忆与 Dream
nanobot 的记忆系统设计为当下保持轻量，长期保持持久。它会注意到哪些值得保留，放下不再需要聚焦的内容，并将真实的经历转化为平静、持久且实用的知识。

记忆层级
nanobot 将记忆分为多个层级，因为不同类型的记忆需要不同的工具：

session.messages
短期记忆。当前聊天窗口中的实时对话。这是 Agent 当前"看到"的内容。发送 /new 时，它会被归档并清除。
memory/history.jsonl
归档。过往对话的压缩摘要，每行一条 JSON。Agent 可以搜索它来查找数周前的事实。
SOUL.md
个性。Bot 的沟通方式——语气、正式程度、语言偏好。当 Dream 发现你的交互模式有一致性时会更新此文件。
USER.md
用户档案。Bot 了解的关于你的信息——姓名、所在地、习惯、偏好。那些你不应该重复说两次的事情。
memory/MEMORY.md
项目知识。做过的决策、配置的工具、架构选择、尝试过但失败的方案。过往会话的集体智慧。
所有这些文件都存储在你的工作区中（默认 ~/.nanobot/workspace/）。你可以直接阅读和编辑它们——都是纯文本 Markdown。

你会体验到什么
在实际使用中，记忆意味着：

你告诉 Bot 你的名字一次。它再也不会问——即使几个月后，即使在不同的聊天渠道中。
你在上午的会话中配置了一个部署策略。晚上，Bot 还记得并遵循同样的方案。
你从 Telegram 切换到终端。Bot 仍然知道你的偏好、你的时区、你的项目上下文。
长对话不会变慢或丢失上下文——较旧的部分会被透明地压缩和归档。
你不需要为此做任何配置。运行 nanobot onboard 后记忆就默认开启了。

Consolidator（整合器）
当对话长到足以对上下文窗口造成压力时（大约当 Prompt Token 超过模型上限的一半），Consolidator 会自动启动。它取出最旧的一批消息，提取关键事实，并将摘要追加到 memory/history.jsonl。

会被提取的内容：

用户事实——个人信息、偏好、表达过的观点
决策——做出的选择、得出的结论
解决方案——通过试错发现的有效方法（尤其是不明显的方法）
事件——计划、截止日期、值得注意的事情
会被跳过的内容：可从源码推导的代码模式、git 历史，或已存在于现有记忆文件中的任何内容。

每条归档条目的格式如下：

{"cursor": 42, "timestamp": "2026-04-03 00:02", "content": "- User prefers dark mode\n- Decided to use PostgreSQL for the auth service"}

这一切都是透明进行的——你不会注意到它。对话正常继续，最旧的轮次在后台被归档。

Auto compact vs 软整合
存在两条整合路径，它们在磁盘上的行为不同。当你以后想回放或审计一段旧会话时，搞清楚跑的是哪一条就很关键。

Token 驱动（软）
前面介绍的 Consolidator。当 prompt token 越过阈值时触发。它只是移动游标——session.last_consolidated——让 LLM 不再看到更早的消息，但磁盘上的 session.messages 保持不动。原始 tool_calls、tool_call_id、reasoning_content 字段全部保留。你可以随时回放完整原始会话。
空闲驱动（硬，idleCompactAfterMinutes）
默认关闭。一旦启用，N 分钟无活动后，会话文件 sessions/<key>.jsonl 会被改写：旧消息被替换为一条摘要，会话的其余部分被覆盖写入。已归档消息里的结构化字段（tool call id、reasoning 块）在物理上消失——只有自然语言摘要会在 memory/history.jsonl 里留下。如果 LLM 摘要失败，nanobot 会退化为一段 [RAW] … 纯文本转储，但仍然会覆盖写会话文件。
!
Auto compact 不是软的游标移动，而是真正的写入。一旦触发，旧轮次的结构化 tool-call 数据就没了。如果你需要回放、审计或调试会话，把 idleCompactAfterMinutes 保持在默认（禁用）即可。

什么时候应该开 auto compact？主要是你在意 KV 缓存复用窗口时——在 KV 缓存过期前强行压缩会话，对那些会对重入上下文收费的 Provider 能实打实省钱。日常用，关着就好。

Dream
Dream 是更慢、更深思熟虑的层级。每隔几个小时，它会读取新的历史条目和当前的长期文件，然后做出最小幅度的诚实编辑，以保持记忆的连贯性和时效性。

Dream 分两个阶段工作：

分析——将新的对话历史与当前记忆文件进行对比。识别新事实、更正和过时条目。同时检测记忆文件之间的冗余，标记重复或陈旧内容。
编辑——精确更新 SOUL.md、USER.md 和 memory/MEMORY.md。从不重写整个文件——只添加、更新或删除特定行。
每次 Dream 运行后，变更会被 commit 到本地 git 仓库。你可以查看具体改了什么：

/dream-log

## Dream Update
- Commit: a3f8c21
- Time: 2026-04-03 14:30
- Changed files: USER.md, memory/MEMORY.md

```diff
--- a/USER.md
+++ b/USER.md
@@ -3,6 +3,7 @@
 - Timezone: America/Chicago
 - Prefers concise responses
+- Recently started using PostgreSQL for new projects
```

Use /dream-restore a3f8c21 to undo this change.

现在 commit body 里会带上完整的 Phase 1 分析文本，所以 /dream-log 不只给你看 Dream 改了什么，也会给你看 为什么改——LLM 的推理痕迹被留在了 git 历史里。

技能发现
Dream 现在可以将重复的工作流提升为 skills/ 目录下的可复用技能。当它检测到某个模式在多个会话中被多次应用时，会将该工作流提取为一个技能文件，Agent 可以在未来的对话中直接调用。

行级 age 信号与去重
Dream 审视 memory/MEMORY.md 时，看到的每一行都带着它的 age——距离这一行上次修改过去了多少天。超过 14 天的行会在行尾加上 ← 37d 这样的标注：

- Decided to use PostgreSQL for the auth service  ← 37d
- Deployment strategy: blue-green via GitHub Actions
- Project uses Bun for the web UI  ← 21d

这些标注只会注入到 Dream 的 prompt 里。你自己打开 MEMORY.md 时看不到任何 ← Nd 标记——磁盘上的文件依旧干净。

SOUL.md 和 USER.md 从不加标注。它们是永久文件——用户的习惯、偏好、性格——age 对它们不是有价值的信号。

Age 是 review 信号，不是删除条件。一条 90 天前但仍然准确的事实会留下。Dream 只会删掉真正过时或错误的行。

Phase 1 的去重意识
现在 Phase 1 会明确做两件事：从近期历史中抽取新事实，以及对现有记忆做去重。Dream 会找四种情况：

同一事实被重复记录在多个地方
可以合并的重叠段落
MEMORY.md 里的条目复述了 USER.md 或 SOUL.md 已有的内容
在不丢信息的前提下可以压缩的冗长条目
一旦 Dream 判断某行是冗余的，会在 Phase 1 就标上 [FILE-REMOVE]，由 Phase 2 干净地删除。

Dream 配置
{
  "agents": {
    "defaults": {
      "dream": {
        "intervalH": 2,              ← Dream 运行间隔（小时）
        "modelOverride": null,       ← 为 Dream 使用不同的模型
        "maxBatchSize": 20,          ← 每次处理的历史条目数
        "maxIterations": 15          ← 编辑阶段的工具预算
      }
    }
  }
}

intervalH
自动 Dream 运行的间隔小时数。默认：2。设大可降低 API 成本，设小可更快更新记忆。
modelOverride
为 Dream 使用不同的（更便宜的）模型。默认：与你的主 Agent 模型相同。一个好策略是用 deepseek-chat 这样的快速模型跑 Dream，而对话保留更强的模型。
maxBatchSize
每次 Dream 处理的新历史条目数量。默认：20。设高一些意味着繁忙一天后追赶进度所需的运行次数更少。
maxIterations
Dream 编辑阶段的工具预算。默认：15（0.1.5.post2 中从 10 上调——在 20 次基准跑里这是经验上最好的权衡点；继续往上走收益递减）。每次迭代是一次 edit_file 调用。
查看和控制 Dream
/dream
立即运行 Dream，而不是等待定时计划。你会看到 "Dream completed." 或 "Dream: nothing to process."
/dream-log
以 git diff 形式查看最新的 Dream 变更——记忆文件中具体添加、修改或删除了哪些行。
/dream-log <sha>
按 commit SHA 查看过去的某次 Dream 变更。
/dream-restore
列出最近约 10 次 Dream commit，包含时间戳和变更文件，供你选择回滚。
/dream-restore <sha>
将记忆回滚到某次变更之前的状态。回滚本身也会被 commit，所以你可以随时撤销这次撤销。
你也可以直接编辑记忆文件——它们只是工作区中的 Markdown 文件。Dream 不会覆盖你的手动修改；它只根据新的对话历史做增补或修正性编辑。
